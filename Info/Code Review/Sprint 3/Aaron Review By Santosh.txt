Readability:
  What is readable:
  The class is very small so it is fairly readable as a whole. It is easy to tell where the fields and public functions are.

  What is less readable:
  There is a long call to SpriteBatch.Draw with various unexplained arguments. In C# it is possible to put the name of a function parameter before the value passed in (Ex: "scale: 1.0f") and I feel this would be more descriptive for a long call like SpriteBatch.Draw.

Code quality:
  The code is very simple and the class has one focus and low cohesion, so the code quality is overall high. However, one comment is that rotation gets set to 0.0f in the constructor, but it doesn't depend on any of the passed in values, so it could be set to 0.0f in the class body where the field is created instead. Similarly for sourceRect.

Hypothetical code change and how well the code supports that change:
  A plausible feature we will probably add in the next sprint is the ability for the locked door types to convert to an unlocked variant. This class would not be suited well for directly converting into an unlocked version, thus any class containing this block (like level class, collision checker possibly) would need to update its data store to create a new block class entirely. This is somewhat messy so a suggestion I have is to make a unified slatted door class that has an enum holding state info like locked or unlocked, and dynamically changes its sprite and oncollide behavior. Then it would be easy to set the door to the unlocked state upon contact with a key.
